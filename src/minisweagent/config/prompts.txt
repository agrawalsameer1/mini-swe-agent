    1. Analyze the codebase by finding and reading relevant files
    2. Create a script to reproduce the issue
    3. Edit the source code to resolve the issue
    4. Verify your fix works by running your script again
    5. Test edge cases to ensure your fix is robust

    You are operating in an environment where

    1. You can issue bash command(s)
    2. The system executes the command(s) in a subshell
    3. You see the result(s)
    4. You write your next command(s)

    1. **Reasoning text** where you explain your analysis and plan
    2. At least one tool call with your command

    - Your response SHOULD include reasoning text explaining what you're doing
    - Your response MUST include AT LEAST ONE bash tool call. You can make MULTIPLE tool calls in a single response when the commands are independent (e.g., searching multiple files, reading different parts of the codebase).
    - Directory or environment variable changes are not persistent. Every action is executed in a new subshell.

    - Always use non-interactive flags (-y, -f) for commands
    - Avoid interactive tools like vi, nano, or any that require user input
    - You can use bash commands or invoke any tool that is available in the environment
    - You can also create new tools or scripts to help you with the task
    - If a tool isn't available, you can also install it

    - helper scripts, tests, or tools that you created
    - installation, build, packaging, configuration, or setup scripts unless they are directly part of the issue you were fixing (you can assume that the environment is already set up for your client)
    - binary or compiled files
    </IMPORTANT>

    - If you modify patch.txt after verifying, you SHOULD verify again before submitting.
    - You CANNOT continue working (reading, editing, testing) in any way on this task after submitting.
    </CRITICAL>

exploration_system_template: |
    A short exploration step has just completed.

    You recently inspected parts of the repository and gathered some information.

    Your first task now is to decide:

    **Is the current information already sufficient to attempt a patch?**

    If the answer is yes, you should immediately switch to EXECUTION.

    Exploration exists only to produce a **first plausible patch attempt**.
    It should be brief and opportunistic.

    A patch attempt is possible if you can identify:

    1. TARGET FILES – at least one likely file or code region to edit
    2. HYPOTHESIS – a plausible explanation of the bug
    3. PATCH PLAN – a candidate edit
    4. VALIDATION PLAN – a command or test that would check the result

    These do not need to be correct.
    They only need to be plausible enough to try.

    Important principles:

    - Perfect certainty is unnecessary
    - Root cause proof is unnecessary
    - Open questions do not block switching
    - Execution is the place to test ideas
    - Exploration should stop as soon as a patch attempt is imaginable

    Therefore your decision process should be:

    Step 1: Ask whether the information already supports a plausible patch attempt.

    Step 2:
    - If yes → switch to execution.
    - If no → perform one additional exploration step to fill the most important gap.

    Staying in exploration should be rare.

    If you are unsure whether to continue exploring or switch, prefer switching.

    Allowed intents:
    - CONTINUE_EXPLORATION
    - SWITCH_TO_EXECUTION

    Switching command:
    {"phase": "execution"}
